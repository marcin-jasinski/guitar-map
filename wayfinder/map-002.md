## Destination

A build-ready spec for a new **Chord progression explorer** tab in the existing app: the
player picks (or edits) a progression in any key and learns **how to solo over it** — the
parent scale/mode it lives in, the per-chord exceptions, the arpeggio shape for each chord,
and how those shapes connect (guide-tone lines, common tones held). The spec covers the
progression data model, the scale/mode inference, the tab layout and chord-stepping
interaction, voicing selection, the preset library, and persistence. Reaching the end means
nothing is left to decide before implementation — this map does not build the tab.

**Reached** — the spec is [`guitar-map/progressions-spec.md`](guitar-map/progressions-spec.md).

## Notes

- Builds on the shipped app (MAP-001 → [`guitar-map/spec.md`](guitar-map/spec.md)); reuse
  `theory.ts`, `Fretboard.svelte`, the favorites store and the existing colour language
  rather than inventing parallel machinery.
- Use `/grilling` and `/domain-modeling` for decision tickets; `/prototype` for the layout
  and the guide-tone/common-tone rendering.
- Settled while charting, binding on every ticket:
  - Core job is **soloing over the changes**, not comping — the fretboard is the main surface.
  - A progression is **Roman numerals and chord symbols together**, either editable, one
    derived from the other via the key.
  - Scale advice is **one parent scale + per-chord exceptions**.
  - Voicings are **auto-picked for smoothness, with manual override**.
  - Connection aids at launch are **guide-tone lines** and **common tones held** only.
  - Display is **one full neck, stepping chord by chord**.
  - **No audio in this tab** — playback stays in the existing tabs, so nothing here depends
    on the audio-engine work.
  - Presets are **curated + freely editable**, and edits can be saved.

## Decisions so far

<!-- one line per closed ticket -->

- [Define the progression data model — numerals, symbols, key](tickets/010-progression-model.md)
  — a progression is a key (root + major|minor) plus an ordered list of Roman numerals; the
  numeral is stored and the symbol always derived via `spell()`, degrees are chromatic
  (1–7 ± ♭/♯) with the 16 shipped chord types and an optional relative `of` for secondary
  function; no bass note, no duration, nothing invalid.
- [Parent scale inference and per-chord exceptions](tickets/012-scale-inference.md) — score the
  9 shipped 7-note scales rooted on the stored tonic by role-weighted chord-tone coverage
  (3rds/7ths double), name the winner modally ("D Dorian"), and state each exception as a
  note swap against the parent ("play A♭ instead of A") under a function label for every
  non-diatonic chord ("secondary dominant of V", "borrowed from C minor"); one plain answer,
  plus a soft line when over half the chords need exceptions.
- [Tab layout and chord-stepping interaction](tickets/013-tab-layout.md) — a left rail holds the
  progression top-to-bottom (numeral, symbol, function label) with the key control and editor
  under it; the neck is whole-neck only, no position window, and it **looks forward** — dashed
  ghosts of the next chord plus a guide-tone line pointing at it. Also introduces the app's first
  tab bar.
- [Smooth voicing selection and manual override](tickets/014-voicing-selection.md) — the voicing
  is never drawn; it is the frame the guide-tone line hangs on, picked by minimising guide-voice
  fret movement (tie-break: hand travel) with one global DP pass over `chordVoicings()`'s
  existing candidates, so overrides are pinned columns and nothing "re-flows"; overrides
  transpose with the key and drop with a soft note when they can't.
- [Render guide-tone lines and held common tones](tickets/015-connection-rendering.md) — guide
  tones draw as two arrowed lines to the next chord (not across the whole progression); a note
  held into the next chord gets a ring cut into its dot, shape rather than hue; all five layers
  default on, independently toggleable, none mutually exclusive, because each owns a different
  visual channel.
- [Curate the preset progression library](tickets/011-preset-library.md) — ten presets, one flat
  list, name only (named for what each teaches, no descriptions or song titles) modelled on
  `PRESET_TUNINGS`; four classics plus six that prove the inference and the exceptions on load.
  A preset stores no root — loading keeps your key's root and switches only its tonality.
- [Persist progressions in the favorites store](tickets/016-progression-persistence.md) — a
  fourth `Content` kind holding key, chords, pinned overrides and step; the store stays at
  `version: 1` (bumping would discard every existing favorite), `window`/`labelMode` go optional,
  and the parent scale, exceptions and auto voicings are recomputed on load rather than stored.
  Presets and saved progressions share one picker under two headings.
- [Bridge a progression chord to the Chord and Arpeggio tabs](tickets/018-tab-bridge.md) — two
  bridges only, placed by scope: "Play this chord" on the rail entry → Chord tab, "Solo on
  <parent>" beside the parent-scale header → Scale tab; no Arpeggio bridge, since the
  progression neck already shows more. `content` becomes per-tab state so the progression
  survives any excursion, and the tab bar is the way back — no breadcrumb.
- [Assemble the build-ready progression explorer spec](tickets/017-assemble-spec.md) — **the
  destination**, written to [`guitar-map/progressions-spec.md`](guitar-map/progressions-spec.md);
  five contradictions between separately-made decisions surfaced and were resolved there, not
  deferred.

## Not yet specified

- Timed, looping backing-track playback with tempo — excluded from this tab for now; revisit
  only if the audio engine gets good enough to make it worth the transport UI.

## Out of scope

- A preset *authoring* format for sharing progressions between users. Was fog pending
  [Persist progressions in the favorites store](tickets/016-progression-persistence.md); that
  resolution puts progressions in localStorage as a `Content` kind with no export path, and the
  app has no backend, no file import/export and no sharing anywhere else. Sharing is a different
  product decision against a redrawn destination, not a step on this route.
- Chromatic approach notes and scale passing-tones as a second "filler notes" layer. Was fog
  pending [Render guide-tone lines and held common tones](tickets/015-connection-rendering.md);
  that resolution fills the last free visual channel on the neck, and the binding charting
  decision is that the launch connection aids are guide-tone lines and common tones **only**.
  A third layer is post-launch work — a fresh effort against a redrawn destination.
- Full chord-scale (per-chord mode, jazz style) display as an alternative to the parent-scale
  view. Was fog pending [Parent scale inference and per-chord exceptions](tickets/012-scale-inference.md);
  that resolution settles exceptions as note-swaps against one parent precisely *because* the
  binding charting decision is one parent scale + exceptions. A per-chord-mode view contradicts
  that decision rather than extending it, so it sits past this destination — a fresh effort if
  it's ever wanted, not a later ticket on this map.
